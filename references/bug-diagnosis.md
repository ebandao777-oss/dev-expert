# Bug诊断 -- 智能Bug诊断

## 输入要求

1. **错误信息**（必填）：错误日志、异常堆栈或报错截图
2. **相关代码**（必填）：出错的代码片段
3. **运行环境**（可选）：操作系统、语言版本、依赖版本
4. **复现步骤**（可选）：如何触发这个错误

## 执行流程

### 第一步：Spec场景检查

诊断前先检查相关spec中的scenario：
- **如有spec**：确认bug是否涉及spec中的scenario，检查scenario的Given/When/Then是否被正确实现
- **spec与实现不一致**：可能是spec理解偏差导致的bug
- **无spec**：诊断后建议补充spec防止同类bug

```
✓ Spec场景检查:
- 相关Scenario: [scenario名称]
- 预期行为: [spec定义]
- 实际行为: [观察到的行为]
- 偏差分析: [spec vs 实现]
```

### 第二步：已知问题检查

检查项目记忆中的已知问题：
- 搜索项目记忆中的类似bug记录
- 检查是否有已知的workaround
- **发现新问题**：诊断完成后提示记录到项目记忆

### 第三步：错误解析

- 识别错误类型（语法错误、运行时错误、逻辑错误、环境错误）
- 提取关键信息（错误码、错误消息、发生位置）
- 分析堆栈跟踪（调用链、最后执行位置）

### 第四步：根因分析

| 错误类型 | 常见根因 | 分析方法 |
|---------|---------|---------|
| **语法错误** | 拼写错误、括号不匹配、缩进错误 | 定位到具体行号和字符 |
| **运行时错误** | 空指针、数组越界、类型错误 | 检查变量状态和数据流 |
| **逻辑错误** | 条件判断错误、算法缺陷 | 追踪执行路径和中间结果 |
| **环境错误** | 依赖缺失、版本不兼容、配置错误 | 检查环境和依赖信息 |
| **并发错误** | 竞态条件、死锁、资源竞争 | 分析线程/协程交互 |

### 第五步：定位问题

- 指出具体的代码位置（文件、函数、行号）
- 说明变量在错误时刻的状态
- 追踪数据流找到问题源头

### 第六步：提供修复（Surgical Changes）

- 给出修复后的代码
- 说明修复原理
- **Karpathy约束**：只修改修复Bug必需的代码，不改相邻代码、风格、注释
- 提供验证修复的方法

### 第七步：定义成功标准（Goal-Driven Execution）

基于已确定的根因和修复方案，明确定义"修复成功"的标准：
- **复现测试**：写一段能稳定触发Bug的测试/步骤
- **成功标准**：修复后应达到什么状态（如"空邮箱不再导致崩溃"）
- **验证方法**：如何确认修复有效（如"运行复现测试10次均通过"）

**示例**：
```
成功标准：用户邮箱为空字符串时，validate_user() 抛出 ValueError 而非崩溃。
复现测试：调用 validate_user({'email': ''}) → 应抛出 ValueError。
验证方法：运行复现测试 → 通过。
```

### 第八步：记录到项目记忆

修复完成后，将关键决策和变更记录到项目记忆（参见 `project-memory-management.md`）：
- 记录 Bug 根因和修复方案（Decision Record 模式）
- 记录新增的预防措施和编码规范（Convention Capture 模式）
- 更新已知问题清单，防止同类 Bug 复发

## 输出格式

```
## Bug诊断报告

**错误类型**：[语法/运行时/逻辑/环境/并发]
**错误信息**：[核心错误消息]
**发生位置**：[文件/函数/行号]

## 根因分析

**问题描述**：[一句话说明]
**详细分析**：[逐步推理过程]
**相关代码**：
```[问题代码]```

## 修复方案

```[修复后的代码]```

**修复说明**：[为什么这样修复]

## 验证方法

1. [步骤1]
2. [步骤2]

## 预防措施

- [建议1]
- [建议2]
```

## 质量标准

- 根因分析必须基于代码和日志证据，不能猜测
- 修复代码必须可运行，且能解决原问题
- 必须提供验证方法，让用户确认修复有效
- 对于不确定的根因，标注"最可能的原因"并列出其他可能性
- 环境相关错误必须说明具体的环境要求或版本限制
- 不得建议用户"重启试试"或"重新安装"作为首要方案
- **Karpathy红线**：修复前必须先写复现测试，修复后验证测试通过
- **Karpathy红线**：只修改修复Bug必需的代码，不改相邻代码、风格、注释

## 关联Skill

- **Karpathy编码规范** — 修复前可用 `Karpathy编码规范` 确认修复策略符合精准修改原则
- **代码审查** — 修复后可用 `代码审查` 检查是否引入新问题
- **测试用例生成** — 修复后可用 `测试用例生成` 生成回归测试
- **代码生成** — 复杂修复可用 `代码生成` 生成替代实现

## 失败回退机制

| 步骤 | 失败条件 | 回退目标 | 最大重试 | 不可恢复时升级路径 |
|------|---------|---------|---------|------------------|
| 第一步：症状收集 | 用户无法提供完整错误信息 | 基于已有信息匹配最接近的错误分类 | 2 | 告知所需最小信息集（错误消息+触发操作），等待补充 |
| 第二步：信息收集 | 项目结构/依赖信息不可用 | 跳过环境探针，仅基于症状和代码分析 | 1 | 标注"环境信息缺失"，可能漏诊环境类Bug |
| 第三步：复现尝试 | 无法稳定复现 | 标注为"间歇性Bug"，记录已观察到的不稳定模式 | 3 | 建议添加日志插桩后重新上报 |
| 第四步：根因分析 | 多种可能根因均符合症状，无法唯一确定 | 列出候选根因，按可能性排序 | 2 | 输出诊断矩阵（候选根因×验证方法），由用户逐一排除 |
| 第五步：定位问题 | 涉及第三方库/外部服务，代码不可审视 | 标注第三方边界，定位调用链最近的可控点 | 1 | 给出workaround方案，建议向第三方提issue |
| 第六步：提供修复 | 修复方案需大规模重构（超出Bug修复范围） | 给出最小修复+重构建议供后续规划 | 2 | 拆分为Bug修复（立即）和架构优化（单独任务） |
| 第七步：定义成功标准 | 修复涉及多系统联调，单次验证不充分 | 定义分层验证标准（单元→集成→端到端） | 1 | 给出回归测试checklist，建议灰度发布验证 |

## 连接器（可选增强）

| 连接器 | 增强能力 |
|--------|---------|
| **Filesystem** | 读取项目源代码和配置文件，定位bug根因 |
| **Git** | 读取提交历史和diff，追溯问题引入时间点 |
| **Notion** | 将诊断报告和修复方案写入 Notion 知识库 |
