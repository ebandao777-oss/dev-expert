# 代码生成 -- 智能代码生成

## 输入要求

1. **功能需求**（必填）：需要实现的功能描述
2. **编程语言**（必填）：如Python、JavaScript、Go、Java
3. **框架/库**（可选）：如React、Spring Boot、FastAPI
4. **约束条件**（可选）：如"时间复杂度O(n)"、"不使用第三方库"
5. **输入输出示例**（可选）：期望的输入和输出格式

## 执行流程

### 第一步：Spec检查

编码前检查是否有明确的spec：
- **如有spec**：提取spec中的scenario作为编码验收依据，严格按Given/When/Then实现
- **如无spec**：提示用户先使用 `Spec驱动开发` 对齐需求，或生成轻量级spec

```
✓ Spec检查: [检测到/未检测到] 需求规格
[如检测到，列出相关scenario]
```

### 第二步：编码前环境确认

生成代码前，先确认环境和依赖：
- 确认目标编程语言和版本
- 确认依赖库和框架版本
- 确认运行环境约束（操作系统、内存、网络等）

**示例**：
```
环境确认：Python 3.10+，无第三方依赖，单文件运行。
```

### 第三步：需求分析

- 理解功能需求和业务场景
- 识别核心算法或设计模式
- 明确约束条件（性能、兼容性、依赖）
- 明确陈述假设：如果需求有歧义，列出多种理解并选择最简方案
- **Karpathy约束**：识别并排除超出需求的功能；如果存在更简单的方案，优先选择简单方案
- **加载项目规范**：从项目记忆中加载相关编码规范（命名约定、错误处理模式等）

### 第四步：设计方案

- 确定函数/类/模块的接口
- 设计数据结构
- 规划错误处理策略
- **Karpathy约束**：最小设计，不预先为"未来可能"做抽象

### 第五步：代码实现（Simplicity First）

生成包含以下要素的代码：
- 清晰的函数/类签名
- 输入验证和参数检查
- 核心逻辑实现（最少代码解决问题）
- 错误处理（只处理可能发生的场景）
- 边界条件处理（空值、越界、超时）
- **简洁优先**（Karpathy原则）：选择最简单的实现方式

**Karpathy约束**：
- 不要超出需求的功能
- 不要为单次使用代码做抽象
- 不要未被要求的"灵活性"或"可配置性"
- 不要为不可能场景做错误处理
- 如果200行可以50行搞定，重写

### 第六步：代码审查

**审查路由规则**：
- **内联精简审查**（本步骤）：适用于单函数/单文件、功能边界清晰、无外部依赖的代码生成，执行以下四项基本检查。
- **完整深度审查**（走 `code-review.md` 完整流程）：适用于多文件/多模块联动、涉及数据持久化/网络通信/安全敏感操作，或代码超过 200 行时，触发 7 步完整审查流程（Spec 一致性→项目规范→代码理解→多维度审查→Karpathy 维度→问题分级→修复建议）。

**内联审查项**：
- 检查代码逻辑是否正确
- 确认是否有更好的实现方式
- 验证边界条件处理
- **精准修改**（Karpathy原则）：只修改必要部分，不引入无关变更

### 第七步：补充说明（Goal-Driven）

- 使用示例
- 复杂度分析
- 注意事项
- **成功标准**：说明如何验证这段代码正确工作

## Task Summary: 代码生成

**完成状态**: [完成/部分完成]
**生成文件**: [文件列表]

**偏差说明**: [与原始需求的差异及原因]
**验证建议**: [如何验证这段代码]
```

## 质量标准

- 生成的代码必须可直接运行，不能是伪代码
- 必须处理常见的边界条件（空输入、越界、类型错误）
- 必须包含错误处理，不能假设输入总是合法
- 变量和函数命名必须符合语言惯例（如Python用snake_case，JS用camelCase）
- 复杂逻辑必须添加注释说明
- 不得生成包含安全漏洞的代码（如SQL拼接、eval执行用户输入）
- **Karpathy红线**：不得为单一需求生成抽象类、接口、配置对象等过度设计

## 关联Skill

- **Karpathy编码规范** — 生成后可用 `Karpathy编码规范` 检查是否符合编码哲学

- **测试用例生成** — 生成后可用 `测试用例生成` 生成配套测试
- **文档生成** — 生成后可用 `文档生成` 生成函数文档

## 失败回退机制

| 步骤 | 失败条件 | 回退目标 | 最大重试 | 不可恢复时升级路径 |
|------|---------|---------|---------|------------------|
| 第一步：场景触发 | 无关联spec/文件/用户上下文 | 直接基于用户描述启动（跳过场景匹配） | 0 | 告知用户缺少上下文，手动描述需求 |
| 第二步：环境确认 | 目标语言/版本无法确认 | 使用当前环境默认设置，标注假设 | 1 | 提示用户明确指定语言和版本 |
| 第三步：需求分析 | 需求存在不可调和的歧义 | 列出歧义点，要求用户澄清 | 0 | 选择最简方案并标注"假设"，由用户后续修正 |
| 第四步：设计方案 | 设计约束冲突（性能vs简洁不可兼得） | 优先简洁方案，标注性能取舍 | 2 | 提供两套方案供用户选择 |
| 第五步：代码实现 | 依赖库不可用或版本不兼容 | 降级到标准库实现或标注兼容性说明 | 2 | 输出伪代码框架+待替换标注，移交人工实现 |
| 第六步：代码审查 | 内联审查发现逻辑缺陷 | 退回第四步重新设计方案 | 2 | 触发 code-review.md 完整审查流程 |
| 第七步：测试验证 | 测试用例无法通过或边缘case遗漏 | 退回第五步修复代码 | 3 | 输出已知问题清单和待修复项，标记"需人工介入" |

## 连接器（可选增强）

| 连接器 | 增强能力 |
|--------|---------|
| **Filesystem** | 读取项目现有代码结构和编码规范，生成风格一致的代码 |
| **Git** | 读取版本历史，了解代码演进上下文 |
| **Notion** | 将生成的代码和设计文档写入 Notion 知识库 |
