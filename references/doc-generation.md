# 文档生成 -- 智能文档生成

**增强能力（连接器加持）**

- ~~Filesystem → 读取整个项目结构，生成完整的项目文档

## 连接器（可选增强）

| 连接器 | 增强能力 |
|--------|---------|
| **~~Filesystem** | 读取项目目录结构和所有源码，生成完整项目文档 |

## 输入要求

1. **代码/项目**（必填）：需要生成文档的代码或项目描述
2. **文档类型**（必填）：函数文档/README/API文档/架构文档
3. **输出格式**（可选）：Markdown/HTML/reStructuredText，默认Markdown
4. **目标读者**（可选）：开发者/用户/运维，默认开发者
5. **语言**（可选）：中文/英文，默认中文

## 执行流程

### 第一步：Spec一致性检查

编写文档前检查与spec的一致性：
- **如有spec**：确保文档描述与spec中的scenario一致
- **发现不一致**：标注文档与spec的偏差，建议同步更新
- **无spec**：文档中标注"基于当前实现，建议补充spec"

```
✓ Spec一致性检查:
- 文档描述 vs Spec: [一致/不一致]
- 需要同步更新: [是/否]
```

### 第二步：内容分析

- 分析目标受众（开发者/用户/维护者）
- 确定文档类型和深度
- 识别关键信息点

### 第三步：结构设计

- 设计文档大纲和章节
- 确定示例和图示
- 规划交叉引用

### 第四步：内容编写

- 编写清晰简洁的说明
- 添加代码示例和截图
- 包含使用场景和最佳实践
- **引用Spec**：在相关章节引用spec中的scenario作为行为依据

### 第五步：质量检查

- 检查准确性和完整性
- 验证代码示例可运行
- 确认格式一致性
- **Spec同步**：确认文档与spec无冲突

## API文档

自动由代码注解/JSDoc/Python docstrings 提取生成。如项目接入 Swagger/OpenAPI，同时导出机器可读接口定义。

## 贡献

[贡献指南]
```

## 质量标准

- 文档必须准确反映代码功能，不能虚构不存在的功能
- 参数说明必须包含类型、必填性和取值范围
- 示例代码必须可直接运行，不能是伪代码
- README必须包含安装和使用说明，不能只有功能列表
- API文档必须包含错误码和异常情况
- 不得遗漏关键信息（如必要的配置项、环境要求）

- **代码生成** — 生成代码后可用 `文档生成` 生成配套文档
